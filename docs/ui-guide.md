# FE 구조 결정 요약 (Cowork 작업용 브리프)

## 프로젝트 컨텍스트
- FE: Next.js (App Router)
- BE: 헥사고날 아키텍처
- 팀 운영: 팀원별로 도메인을 정해서 FE+BE를 함께 담당 (도메인 단위 풀스택 개발)

## 논의 결과 (결정 사항)
1. **FE에는 풀 헥사고날(4계층: ui/application/domain/infra)을 적용하지 않기로 결정.**
   - 이유: FE 도메인 로직은 대부분 얇고(화면 상태/검증/표시 변환 수준), 어댑터 교체 시나리오가 거의 없으며, Next.js 프레임워크 자체가 데이터 패칭과 화면을 가깝게 두는 방식을 권장하기 때문.
2. **대신 두 가지만 지키기로 합의:**
   - 도메인별 폴더 분리 (팀원 담당 단위와 일치)
   - API 호출을 컴포넌트에서 분리 (어댑터 역할의 얇은 한 층만 유지)
3. **App Router의 라우트 폴더를 그대로 도메인 폴더로 사용** (별도 `domains/` 최상위 폴더 불필요, colocation 활용).

## 최종 폴더 구조
```
app/
  orders/                  # 도메인 = 라우트 폴더
    page.tsx               # 화면 진입점 (컴포넌트 조합만)
    components/             # 화면 전용 UI 컴포넌트
    api.ts                  # 백엔드 호출 함수 모음
    types.ts                # 요청/응답 타입

components/                 # 도메인 공통 UI (버튼, 인풋 등)
lib/
  fetcher.ts                # axios 등 공통 통신 설정
```

## 핵심 규칙 (1줄)
> 컴포넌트에서 직접 API를 호출하지 말고, 같은 도메인 폴더의 `api.ts`를 거쳐서 호출한다.

## 확장 기준
대부분 도메인은 위 구조로 충분. 단, 가격 계산/멀티스텝 폼/낙관적 업데이트처럼 **클라이언트 로직이 실제로 복잡한 도메인**에만 `logic.ts` 등 파일을 추가로 분리.
